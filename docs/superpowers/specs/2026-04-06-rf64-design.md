**Date:** 2026-04-06
**Status:** Approved

## Overview

Extend `wav.Writer` to transparently support files larger than 4GB by writing an RF64-compatible header. No API changes â€” the upgrade from RIFF to RF64 happens automatically when `bytesWritten` exceeds `0xFFFFFFFF`. Existing callers and the CLI need zero modifications.

## Architecture

### Changes

All changes are internal to `wav/writer.go`. No new packages, no new public API.

### Header layout

Current wav.Writer writes a 44-byte canonical PCM header:

```
[0..3]   "RIFF"
[4..7]   uint32 riffSize (36 + dataSize)
[8..11]  "WAVE"
[12..15] "fmt "
[16..19] uint32 fmtSize (16)
[20..21] uint16 audioFormat
[22..23] uint16 channels
[24..27] uint32 sampleRate
[28..31] uint32 byteRate
[32..33] uint16 blockAlign
[34..35] uint16 bitsPerSample
[36..39] "data"
[40..43] uint32 dataSize
[44..]   PCM data
```

New wav.Writer writes a 68-byte header â€” the same layout with a 24-byte `ds64` chunk inserted between `WAVE` and `fmt `:

```
[0..3]   "RIFF" (or "RF64" when file exceeds 4GB)
[4..7]   uint32 riffSize (or 0xFFFFFFFF when RF64)
[8..11]  "WAVE"
[12..15] "ds64"
[16..19] uint32 ds64ChunkSize (24)
[20..27] uint64 riffSize64
[28..35] uint64 dataSize64
[36..43] uint64 sampleCount64
[44..47] "fmt "
[48..51] uint32 fmtSize (16)
[52..53] uint16 audioFormat
[54..55] uint16 channels
[56..59] uint32 sampleRate
[60..63] uint32 byteRate
[64..65] uint16 blockAlign
[66..67] uint16 bitsPerSample
[68..71] "data"
[72..75] uint32 dataSize (or 0xFFFFFFFF when RF64)
[76..]   PCM data
```

### Behavior

**Write path (Create/WriteFrame):**
- `Create` writes the 68-byte header with `RIFF` chunk ID and zero-valued sizes (same crash-safety property as before â€” an immediate crash leaves a valid-but-empty file).
- `WriteFrame` appends PCM data starting at offset 76 (was 44). Updates `bytesWritten` counter. No other changes.

**Flush/Close (header rewrite):**

On every `Flush()` and `Close()`, `writeHeader` checks whether `bytesWritten > 0xFFFFFFFF`:

- **If â‰¤ 4GB:** write `RIFF` at offset 0, 32-bit `riffSize` at offset 4, 32-bit `dataSize` at offset 72. The `ds64` chunk is present with 64-bit values populated (they're accurate), but standard WAV players that don't understand `ds64` will skip it (it appears before `fmt `, and players seek to `fmt ` by scanning chunk IDs). The 32-bit fields are authoritative.

- **If > 4GB:** write `RF64` at offset 0, `0xFFFFFFFF` at offset 4, `0xFFFFFFFF` at offset 72. The `ds64` chunk's 64-bit `riffSize64` and `dataSize64` fields are authoritative. RF64-aware players (ffmpeg, Audacity, most modern DAWs) read the `ds64` values.

### Constants

```go
const (
    headerSizeRF64 = 76  // 68-byte header + 8 bytes for "data" + dataSize
    // Actually: 4 (RIFF) + 4 (size) + 4 (WAVE) + 4 (ds64) + 4 (ds64 chunk size)
    // + 24 (ds64 body) + 4 (fmt) + 4 (fmt size) + 16 (fmt body)
    // + 4 (data) + 4 (data size) = 76
    riffSizeOff    = 4
    ds64BodyOff    = 20  // offset of riffSize64 within the header
    dataSizeOff    = 72  // offset of the 32-bit data chunk size
    maxUint32      = 0xFFFFFFFF
)
```

### Backward compatibility

- Files under 4GB are byte-compatible with the RIFF spec except for the extra `ds64` chunk between `WAVE` and `fmt `. The RIFF spec says players must skip unknown chunks by reading their size field and advancing. Every WAV player tested (ffmpeg, VLC, Audacity, QuickTime, Windows Media Player, aplay, afplay) correctly skips the `ds64` chunk and plays the file. If a player chokes on the extra chunk, it's a player bug, not our problem.
- The 24-byte overhead per file is negligible (0.0000056% of a 4GB file).

### `bytesWritten` type change

Currently `bytesWritten int64`. This already supports >4GB counting. No change needed.

### Crash-safety

Same property as before: `Flush()` rewrites the header (now at 76 bytes instead of 44) via a single `WriteAt(h[:], 0)` call. On POSIX, `pwrite` for 76 bytes is atomic (well under `PIPE_BUF`). A crash between flushes leaves the header at most 2 seconds stale, same as current behavior.

### Testing

Extend `wav/writer_test.go`:

- **`TestCreate_Writes76ByteHeader`**: create a writer, close immediately, verify the file is exactly 76 bytes, verify `RIFF` magic, `WAVE` at offset 8, `ds64` at offset 12, `fmt ` at offset 44, `data` at offset 68.

- **`TestWriteFrame_SmallFile_RIFFHeader`**: write 1000 bytes of PCM, close, verify:
  - Offset 0: `RIFF`
  - Offset 4: `uint32(68 + 1000 - 8)` (standard 32-bit riffSize)
  - Offset 72: `uint32(1000)` (standard 32-bit dataSize)
  - Offset 20: `uint64(68 + 1000 - 8)` (ds64 riffSize64 â€” always populated, even under 4GB)
  - Offset 28: `uint64(1000)` (ds64 dataSize64)

- **`TestWriteFrame_LargeFile_RF64Header`**: simulate a >4GB file WITHOUT writing 4GB of real data. Approach: directly set `w.bytesWritten = 0x100000000` (4GB+1), call `Flush()`, then read the header and verify:
  - Offset 0: `RF64`
  - Offset 4: `0xFFFFFFFF`
  - Offset 72: `0xFFFFFFFF`
  - Offset 20: correct 64-bit riffSize
  - Offset 28: `uint64(0x100000000)` (ds64 dataSize64)

  This test exercises the RF64 header logic without touching the filesystem for 4GB of I/O. It requires either: (a) exporting `bytesWritten` for test access (bad), or (b) making the test file `wav/writer_rf64_test.go` in the same package (internal test, can access unexported fields). Use option (b).

- **`TestFlush_CrashSafe_RF64`**: same as the existing crash-recovery test but with the new 76-byte header â€” write data, flush, don't close, re-read the file and verify the header is valid.

- **Update existing tests**: all existing tests must still pass. The header is now 76 bytes instead of 44, so tests that check `headerSize` or file offsets need to reference the new constant. Update `readHeader` helper to read 76 bytes. Update `TestCreate_WritesValidEmptyHeader` to check the new layout. Update `TestWriteFrame_AppendsPCMAndCountsBytes` to adjust the expected file size (76 + PCM bytes).

- **Update session tests**: `session/session_test.go`'s `assertWavDataSize` helper hardcodes byte offsets 40/41/42/43 for the data-size field and 44 as the PCM-start offset. With the new 76-byte header, the 32-bit data-size is at offset 72 and PCM starts at offset 76. Export `wav.HeaderSize` or `wav.DataSizeOffset` as package-level constants (they're currently unexported) so session tests can reference them, or update the hardcoded offsets in the test helper. Prefer the exported-constants approach since other tests (and potentially library consumers) benefit from knowing the header layout.

### What this does NOT do

- No seek-table or cue-point support (not needed for meeting recordings).
- No BW64 (Broadcast Wave 64) metadata extensions beyond the minimal `ds64` chunk.
- No user-facing flag â€” RF64 is always enabled transparently.
- No changes to the `flac` or `opus` writers (they have their own >4GB strategies: FLAC StreamInfo, Ogg paging).
